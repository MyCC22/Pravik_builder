# Booking / Lead Capture Tool — Design Spec

## Overview

A modular booking/lead capture form that business owners can attach to any generated website. Customers click a CTA ("Book a Trial", "Get a Quote", etc.) and land on a standalone form page. The form fields are defined as structured JSON — generated by AI based on the business type, editable via chat/voice commands. Submissions are stored in Supabase.

## Goals

- **Reliable form rendering** — JSON schema drives the form deterministically; AI never generates raw HTML for the form
- **Modular** — works with any template/theme; standalone page at `/book/{projectId}`
- **AI-native editing** — business owners modify fields through the same chat/voice interface they use for website editing
- **Theme-matched** — form page inherits the site's theme (clean/bold/vibrant/warm)

## Non-Goals (V1)

- Submissions dashboard UI
- Email/text notifications on new submissions
- Drag-and-drop visual form builder
- File upload fields
- Multi-step forms
- Conditional field logic
- Form analytics

---

## Architecture

```
User says "Build me a soccer website"
  → generateSite() builds website (existing flow)
  → generateBookingTool() creates tool config JSON (new)
  → Config stored in `tools` table
  → Generator sets bookingUrl on template_config → CTA buttons auto-link to /book/{projectId}

User says "Add a field for shirt size"
  → Router classifies as `edit_tool` intent (new)
  → Orchestrator loads tool config from DB
  → ToolEditor agent modifies JSON config
  → Updated config saved to DB

Customer visits /book/{projectId}
  → Next.js page fetches tool config + project theme
  → React renders split layout: branding panel + form
  → Submit → POST /api/tools/submit → tool_submissions table
  → Thank-you message shown
```

## Data Model

### New table: `tools`

| Column | Type | Description |
|--------|------|-------------|
| id | UUID (PK) | Auto-generated |
| project_id | UUID → projects(id) | Parent project |
| tool_type | TEXT | Tool category, e.g. `'booking'` |
| config | JSONB | Field definitions + branding |
| is_active | BOOLEAN | Default `true` |
| created_at | TIMESTAMPTZ | Auto |
| updated_at | TIMESTAMPTZ | Auto |

### New table: `tool_submissions`

| Column | Type | Description |
|--------|------|-------------|
| id | UUID (PK) | Auto-generated |
| tool_id | UUID → tools(id) | Parent tool |
| data | JSONB | Submitted form data |
| submitted_at | TIMESTAMPTZ | Auto |

### RLS Policies

**`tools` table:**
- `SELECT`: Allow all (public) — customers need to read tool config to render the form
- `INSERT`: Allow when `auth.uid() = (SELECT user_id FROM projects WHERE id = project_id)` — only project owner can create tools
- `UPDATE`: Same as INSERT — only project owner can modify
- `DELETE`: Same as INSERT — only project owner can delete

**`tool_submissions` table:**
- `INSERT`: Allow all (public) — unauthenticated customers submit forms
- `SELECT`: Allow when `auth.uid() = (SELECT p.user_id FROM projects p JOIN tools t ON t.project_id = p.id WHERE t.id = tool_id)` — only project owner can read submissions
- `UPDATE`: Deny all — submissions are immutable
- `DELETE`: Allow same as SELECT — only project owner

**Note:** The submit API endpoint (`/api/tools/submit`) uses the Supabase service role key to bypass RLS for inserts, since form submissions come from unauthenticated customers. Server-side validation acts as the security boundary.

### Supported Field Types (V1)

| Type | Renders As | Validation |
|------|-----------|------------|
| `text` | `<input type="text">` | Required check; max 500 chars |
| `email` | `<input type="email">` | Email format regex (`/.+@.+\..+/`) |
| `phone` | `<input type="tel">` | Must contain at least 7 digits (stripped of formatting chars) |
| `textarea` | `<textarea>` | Required check; max 2000 chars |
| `number` | `<input type="number">` | Must be numeric |
| `dropdown` | `<select>` + `<option>` | Required = must select non-empty value; value must be in `options` array |

## AI Agents

### ToolGenerator (new)

- **Model:** Claude Haiku — sufficient for JSON generation from a clear schema; keeps cost low and latency under 2s
- **Purpose:** Generate initial booking tool config from business description
- **Input:** Business description (from the original site generation prompt) + template type
- **Output:** Complete `ToolConfig` JSON
- **When called:** Automatically after `generateSite()` completes (both initial generation and rebuilds)
- **Failure behavior:** If `generateBookingTool()` fails, log the error and continue — the site is still valid without a booking tool. The user can add one later via `add_tool` intent.

### ToolEditor (new)

- **Model:** Claude Haiku — JSON-in, JSON-out editing is a simple task
- **Purpose:** Modify tool config based on user's chat/voice command
- **Input:** Current `ToolConfig` JSON + user message
- **Output:** Updated `ToolConfig` JSON
- **When called:** When router classifies intent as `edit_tool`

### Router Changes

New intents added to `AgentIntent`:

- `edit_tool` — User wants to modify booking form fields, branding, or text
- `add_tool` — User wants to add a booking form (when one doesn't exist yet)

**Intent priority / disambiguation rules:**
- "add a booking form" / "add a contact form" / "add lead capture" → `add_tool` (new tool, not a block)
- "add a field for X" / "change the form" / "update the dropdown" / "make email required" → `edit_tool` (modify existing tool)
- "add a contact section" / "add a features section" → `add_block` (HTML section, not tool)
- The key signal is **form-specific language** (fields, dropdown, required, form, booking form) → tool intents. **Section-specific language** (section, block) → block intents.

### AgentResponse Changes

New `action` values added to the `AgentResponse` union:

```typescript
action: 'generated' | 'edited' | 'theme_changed' | 'removed' | 'reordered' | 'clarify'
       | 'tool_created' | 'tool_edited'
```

- `tool_created` — returned when a new booking tool is generated (via `add_tool` or auto-generation)
- `tool_edited` — returned when tool config is modified (via `edit_tool`)

The chat panel renders these the same as `edited` — a success message. The builder page does NOT need to refresh the preview iframe for tool actions (the tool lives on a separate page).

### Orchestrator Changes

New cases in the switch statement:

**`edit_tool`:**
1. Load tool from `tools` table for this project
2. If no tool exists → return `clarify` with "You don't have a booking form yet. Want me to create one?"
3. Pass `tool.config` + user message to ToolEditor agent
4. Validate returned JSON matches `ToolConfig` shape
5. Save updated config back to `tools` table
6. Return `{ action: 'tool_edited', message: "Done! I updated..." }`

**`add_tool`:**
1. Check if a booking tool already exists for this project
2. If exists → return `clarify` with "You already have a booking form. Would you like to edit it?"
3. If not → call ToolGenerator with business context (project name + original description)
4. Save new tool config to `tools` table
5. Update `bookingUrl` on project's `template_config` to `/book/{projectId}`
6. Return `{ action: 'tool_created', message: "Done! I created a booking form..." }`

### CTA Integration with Existing Templates

The `TemplateContent` type already has `bookingUrl` and `bookingText` fields. When a booking tool is created:

1. `generateBookingTool()` (or `add_tool` handler) updates the project's `template_config.content.bookingUrl` to `/book/{projectId}`
2. `template_config.content.bookingText` is set from `toolConfig.submitText` (e.g. "Book My Free Trial")
3. Existing template sections (hero CTA, booking-cta section) already read these fields → buttons auto-link to the form page
4. On rebuild (`generate_site` intent): existing tool is deleted and regenerated fresh (the tool config is tightly coupled to the business description)

## Form Page

### Route

`/book/[projectId]` — standalone page, no auth required (public-facing for customers)

### Layout

Split layout:
- **Left panel:** Business name, tool title, subtitle, trust signal checkmarks. Uses theme accent colors as background.
- **Right panel:** Form fields rendered from JSON config. White/surface background.
- **Mobile:** Stacks vertically — condensed branding on top, form below.

### Rendering Pipeline

1. Server component fetches tool config + project (theme, name) from Supabase
2. Client component receives props, renders split layout
3. Form renderer maps each field in `config.fields` to a React component
4. Client-side validation on submit (required fields, email format, phone digits, max lengths)
5. `POST /api/tools/submit` with `{ tool_id, data }` body
6. On 200 → swap form for thank-you message from `config.successMessage`
7. On error → show inline error, let user retry

### Error States

| Condition | Behavior |
|-----------|----------|
| `projectId` not found | Show 404 page: "This page doesn't exist" |
| Tool exists but `is_active` is false | Show message: "This form is currently unavailable" |
| Tool config is malformed / missing fields array | Show message: "This form is being set up. Please check back soon." |
| No tool exists for this project | Show message: "No booking form has been set up for this site yet" |

### Theme Matching

Uses `getThemeClasses(themeId)` to pull the project's Tailwind theme:
- Left panel background: `accentBg` / dark variant
- Trust signal checks: `accent` color
- Form surface: `surface` / `bg`
- Button: `accentBg` + `accentText`
- Text: `text` / `textMuted`

Page wrapped in same Tailwind shell (CDN, Inter font) as the generated website.

### Preview in Builder

The booking form is NOT shown in the builder's preview iframe — it lives on a separate page. The builder chat responds with a link: "Your booking form is live at /book/{projectId}" when a tool is created or edited. Future enhancement: add a "View Form" button to the builder UI.

## API Endpoint

### `POST /api/tools/submit`

**Request:**
```json
{
  "tool_id": "uuid",
  "data": {
    "child_name": "Max",
    "age_group": "8-10",
    "experience": "Beginner",
    "parent_email": "parent@example.com",
    "phone": "(555) 123-4567"
  }
}
```

**Response (200):**
```json
{ "success": true }
```

**Response (429):**
```json
{ "error": "Too many submissions. Please try again later." }
```

**Server-side validation:**
1. `tool_id` must exist and `is_active` must be true
2. Load tool config from DB, validate `data` against `config.fields`:
   - All required fields must be present and non-empty
   - Email fields: regex validation
   - Phone fields: at least 7 digits
   - Dropdown fields: value must be in the `options` array
   - All string values: max 2000 characters
3. Strip HTML tags from all string values (XSS prevention)
4. Total `data` payload must be under 50KB

**Rate limiting (V1):**
- IP-based rate limit: max 10 submissions per IP per minute
- Per-tool rate limit: max 100 submissions per tool per hour
- Implemented via in-memory rate limiter in the API route (simple Map with TTL cleanup)
- On rate limit exceeded: return 429

**Security:** Uses Supabase service role key for the insert (since customers are unauthenticated). Server-side validation is the primary security boundary — the endpoint validates all data before inserting.
